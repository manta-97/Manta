# 🐟 Manta — Manifesto v2

## 1. 우리는 무엇을 만드는가

> **Manta는 작업을 파일로 정의하고,**
> **AI와 사람이 같은 규칙으로 조작할 수 있게 만드는**
> **로컬 작업 운영 계층이다.**

우리는 단순한 task 관리 앱을 만들지 않는다.
우리는 단순히 AI가 폴더를 만들고 정리하는 도구도 만들지 않는다.

우리는 **작업을 다루는 방식 자체를 다시 정의**한다.
정확히는, 작업을 저장하는 앱이 아니라
**작업이 존재하고, 이동하고, 검증되고, 추적되는 표준 구조**를 만든다.

---

## 2. 우리가 문제라고 보는 것

현재의 작업 관리 도구들은 공통적으로 같은 한계를 가진다:

* 작업이 데이터베이스에 갇혀 있다
* 시스템은 서버 중심으로 설계되어 있다
* 인터페이스는 GUI 중심이다
* 자동화와 AI가 자연스럽게 개입하기 어렵다
* 작업의 상태와 문맥이 여러 도구에 흩어진다

이 구조에서는:

* 데이터는 사용자 것이 아니다
* 작업은 사람이 직접 조작해야 한다
* AI는 주변적인 도구에 머문다
* 자동화는 일회성 스크립트에 의존한다
* 시스템을 바꿀수록 작업 자산이 사라진다

그리고 반대편에는 또 다른 문제가 있다.

AI에게 “폴더 구조를 만들어라”, “Markdown으로 정리해라”라고 시키는 방식은
빠르게 시작할 수는 있어도 결국 다음 문제를 남긴다:

* 구조가 매번 달라진다
* 상태 전환 규칙이 없다
* 사람이 개입할 기준이 모호하다
* 복구와 검증이 어렵다
* 장기적으로 신뢰 가능한 운영 방식이 되지 못한다

우리는 이 구조를 근본적으로 바꾼다.

---

## 3. 우리의 핵심 신념

---

### 3.1 작업은 파일이다

> **작업은 DB 레코드가 아니라 파일로 존재해야 한다**

파일은 가장 단순하고 강력한 단위다:

* 사람이 읽을 수 있다
* AI가 이해할 수 있다
* Git으로 추적할 수 있다
* 시스템 간 이동이 가능하다

우리는 작업을 “데이터”가 아니라
**“존재하는 객체”로 다룬다.**

중요한 것은 파일 자체보다,
**파일이 시스템의 진실 원본이 된다는 점**이다.

---

### 3.2 SQLite는 로컬 운영 엔진이다

> **모든 것을 Markdown에만 넣는 것은 목표가 아니다**

Markdown은 작업의 장기 원본이다.
하지만 검색, 목록, 필터, context 후보 추출, GUI 상태 같은 운영 경험은
파일만으로는 충분히 강하지 않다.

Manta는 사용자 홈의 root SQLite를 로컬 작업 운영 엔진으로 사용한다.

```text
~/.manta/manta.sqlite
```

이 DB는 서버 원본이 아니다.
사용자의 컴퓨터 안에 있는 로컬 엔진이며,
여러 프로젝트를 빠르게 찾고 조립하기 위한 계층이다.

각 프로젝트 안에는 작은 연결점만 둔다.

```text
.manta/project.json
```

이 파일은 프로젝트의 영구 ID를 가진다.
프로젝트 폴더가 이동되어도 Manta는 이 ID로 root SQLite의 기존 기록과 다시 연결한다.

원칙은 단순하다:

* 작업 본문, 상태, 결정, 결과는 파일이 원본이다
* root SQLite는 검색, 인덱스, ranking, 로컬 UI 메타데이터를 맡는다
* root SQLite는 삭제되어도 프로젝트 파일과 anchor에서 다시 만들 수 있어야 한다
* root SQLite는 Manta 프로그램을 써야 하는 이유를 만든다

즉, Manta는 Markdown 폴더 규약에 머무르지 않는다.
Manta는 파일 위에 빠른 로컬 운영 엔진을 얹는다.

---

### 3.3 단순함이 가장 강력하다

> **우리는 복잡한 기능보다 단순한 구조를 선택한다**

우리는 다음을 거부한다:

* 복잡한 워크플로우
* 과도한 커스터마이징
* 설정 중심의 시스템

대신 우리는 선택한다:

* 명확한 상태
* 예측 가능한 동작
* 최소한의 규칙
* 누구나 같은 방식으로 해석되는 구조

> 단순함은 제약이 아니라 확장성의 기반이다

---

### 3.4 CLI는 시스템의 중심이다

> **작업 시스템은 프로그램이 사용할 수 있어야 한다**

GUI는 사람을 위한 인터페이스다.
CLI는 시스템을 위한 인터페이스다.

Manta에서:

* CLI는 핵심 인터페이스다
* UI는 그 위에 올라가는 레이어다

이 구조를 통해 우리는:

* 자동화
* 스크립팅
* 시스템 간 연결

을 가능하게 한다.

CLI는 편의 기능이 아니라
**작업 시스템의 canonical API**다.

---

### 3.5 AI는 사용자다

> **AI는 도구가 아니라, 시스템의 직접 사용자다**

우리는 AI를 다음과 같이 본다:

* 작업을 생성할 수 있는 존재
* 작업을 수정할 수 있는 존재
* 상태를 관리할 수 있는 존재

이를 위해 Manta는:

* 구조화된 데이터
* 명확한 상태 모델
* 결정적인 인터페이스
* 검증 가능한 작업 규약

를 제공한다.

> Manta는 인간과 AI가 같은 시스템을 공유하는 환경이다.
> 동시에, 같은 규칙을 따라야 하는 환경이다.

우리는 AI가 “똑똑하게 알아서 해주길” 기대하지 않는다.
우리는 AI가 **항상 같은 방식으로 일할 수 있게** 만든다.

---

### 3.6 데이터는 로컬에 있어야 한다

> **작업 데이터는 사용자가 완전히 소유해야 한다**

* 클라우드에 종속되지 않는다
* 서버에 묶이지 않는다
* 언제든 이동 가능하다

Manta는 데이터를 “보관”하지 않는다.
**사용자가 직접 관리할 수 있게 한다**

데이터 소유권은 철학적 취향이 아니라
복구 가능성과 이식 가능성의 조건이다.

작업은 언제든:

* 직접 열어볼 수 있어야 하고
* 수정할 수 있어야 하며
* Git으로 되돌릴 수 있어야 하고
* 다른 도구로 옮길 수 있어야 한다

그래야 AI가 개입해도 시스템을 신뢰할 수 있다.

---

### 3.7 작업은 규약이어야 한다

> **중요한 것은 파일 포맷이 아니라,**
> **작업을 다루는 공통 규약이다**

우리가 만들려는 가치는
“AI가 폴더 구조를 대신 만든다”가 아니다.

그것은 쉽게 복제된다.

우리의 가치는 다음에 있다:

* 작업 생성 방식이 일정하다
* 상태 전환 규칙이 명확하다
* 결정과 산출물이 연결된다
* 사람이 언제 개입해야 하는지 분명하다
* AI가 잘못 다뤄도 복구할 수 있다

즉, Manta는 단순한 파일 묶음이 아니라
**작업을 신뢰 가능하게 다루기 위한 운영 규약**이다.

## 4. 우리가 지향하는 구조

Manta는 다음과 같은 구조를 가진다:

* 작업은 파일로 존재한다
* 시스템은 로컬에서 동작한다
* root SQLite는 빠른 검색과 운영 메타데이터를 담당한다
* 프로젝트 anchor는 폴더 이동 후에도 같은 프로젝트를 식별한다
* CLI가 중심 인터페이스다
* UI는 보조 레이어다
* 작업 상태는 명확한 모델을 가진다
* AI와 사람은 같은 규약을 사용한다

이 구조는:

* 사람이 직관적으로 사용할 수 있고
* AI가 직접 조작할 수 있으며
* 시스템이 단순하게 유지되도록 하며
* 자동화가 누적 가능한 방식으로 작동하게 한다

우리가 만드는 것은 task viewer가 아니다.
우리가 만드는 것은
**작업을 생성하고, 진행시키고, 닫는 방식을 표준화하는 계층**이다.

---

## 5. 우리가 가져온 것, 그리고 다른 점

우리는 기존 도구에서 배웠다.

* **Linear**
  → 단순함, 빠른 조작, 최소 상태

* **Obsidian**
  → 파일 기반, 로컬-first, 데이터 소유권

하지만 우리는 다르게 간다.

> 우리는 이 둘을 결합하고,
> **AI가 직접 사용하는 시스템으로 확장한다**

그리고 여기서 한 걸음 더 나간다.

Jira나 기존 협업 툴에 AI를 연결하는 방식은
기존 시스템 위에 AI를 얹는 접근이다.

Manta는 반대로,
처음부터 AI와 사람이 함께 사용할 수 있는
**작업의 기본 구조 자체**를 만든다.

우리는 기존 도구를 대체하기보다,
그 아래에서 작동하는 더 근본적인 레이어를 지향한다.

---

## 6. 우리가 가지 않는 길

우리는 다음을 선택하지 않는다:

* 복잡한 워크플로우 엔진
* 서버 DB나 원본 DB에 작업을 가두는 설계
* GUI 중심 제품 구조
* 초기부터 협업 중심 설계
* AI의 비결정적 행동에 의존하는 UX
* 일회성 자동화 스크립트에 기대는 운영

이것들은 확장을 어렵게 만들고,
시스템을 무겁게 만든다.

우리는 “AI가 잘 알아서 정리해주는 느낌”보다
**AI가 믿을 수 있는 규칙 안에서 일하는 구조**를 선택한다.

---

## 7. 우리가 가려는 방향

Manta는 여기서 멈추지 않는다.

> **우리는 파일 기반 작업 시스템에**
> **신뢰 가능한 Sync를 결합하려 한다**

---

### 7.1 Sync에 대한 우리의 생각

Sync는 단순한 기능이 아니다.
이 시스템의 가장 어려운 문제다.

우리는 다음을 목표로 한다:

* 파일 기반 구조를 유지하면서
* 여러 환경에서 동일한 상태를 공유하고
* 충돌을 이해 가능한 방식으로 해결한다

---

### 7.2 우리가 지키는 원칙

> **Sync는 단순함을 깨지 않아야 한다**

* 파일 구조를 유지한다
* 사용자 제어를 유지한다
* 시스템을 복잡하게 만들지 않는다

---

## 8. 우리가 만들고 싶은 세계

> 작업은 파일로 존재하고
> AI와 사람이 함께 그것을 조작하며
> 어디서든 동일한 상태를 유지하고
> 누구나 같은 규칙으로 해석할 수 있는 시스템

---

## 9. 최종 정의

> **Manta는 파일 기반 작업 관리 도구가 아니다**
>
> **Manta는 AI와 사람이 같은 규칙으로 작업을 생성, 변경, 추적, 완료할 수 있게 만드는**
> **로컬 작업 운영 계층이다**

---

이 문서는 네가 계속 참고해야 할 기준이다.

앞으로 어떤 결정을 하든:

* “이게 파일 기반 철학을 깨는가?”
* “이게 단순함을 해치는가?”
* “이게 AI 사용성을 떨어뜨리는가?”
* “이게 작업 규약의 일관성을 약하게 만드는가?”
* “이게 복구 가능성과 소유권을 해치는가?”

이 다섯 가지만 보면 된다.
