## 문제 정의

Manta는 Jira나 Notion 연동 도구가 아니다.
Manta는 AI를 많이 쓰는 1인 소프트웨어 작업의 source of truth가 되어야 한다.

목표 사용자는 혼자 일하거나 프리랜서/인디 맥락에서 일하고, Codex나 Claude를 많이 쓰며, 회사처럼 태스크 히스토리를 남기고 싶지만 무거운 회사식 워크플로는 들이고 싶지 않은 사람이다.

핵심 고통은 "태스크 앱이 하나 더 필요하다"가 아니다.
진짜 고통은 개발자가 로컬 Markdown 기록을 직접 관리하지 않으면, AI 보조 작업의 의도가 세션, 태스크, 이후 리뷰 과정에서 쉽게 사라진다는 점이다.

Manta는 local-first, AI-native 태스크 시스템으로 시작한다.
태스크가 파일로 존재하고, 터미널에서 빠르게 관리할 수 있으며, 나중에 AI 세션의 컨텍스트로 다시 조립할 수 있는 가벼운 Linear 같은 제품이어야 한다.

다만 Manta가 단순 Markdown 폴더 규약에만 머물면 제품의 이유가 약해진다.
Markdown은 source of truth로 남기되, Manta는 각 프로젝트 안에 로컬 SQLite 인덱스를 만들어 검색, 목록, 상태 조회, context 후보 추출, GUI 표시를 빠르게 처리해야 한다.
사용자는 파일을 소유하고, Manta는 그 파일 위에 빠른 작업 운영 엔진을 제공한다.

## 제품 원칙

### CLI-first, not CLI-only

CLI는 Manta의 표준 API다.
사람, 스크립트, AI 에이전트가 모두 같은 명령어를 사용할 수 있어야 한다.

하지만 비즈니스는 CLI-only가 되면 약하다.
Manta는 CLI-first 제품이어야 하고, GUI는 무료 제품의 일부로 제공되어야 한다.

### GUI는 무료 Local Workspace

GUI는 유료 잠금 기능이 아니다.
한 대의 로컬 머신에서 쓰는 1인 빌더에게는 무료 제품만으로 완결된 느낌이어야 한다.

GUI의 역할:

- 현재 작업을 빠르게 본다.
- 과거 작업을 빠르게 찾는다.
- Markdown task를 편집한다.
- 필요할 때 AI context를 생성한다.

GUI는 Jira식 관리자 대시보드가 아니다.
가벼운 Linear + Markdown editor + on-demand AI context action에 가깝다.

### CLI와 GUI는 같은 core를 쓴다

```
          ┌─────────────┐
CLI   ──▶ │             │
          │ @manta/core │ ──▶ local files
GUI   ──▶ │             │
          └─────────────┘
                  │
                  ▼
          derived SQLite index
```

원칙:

- CLI와 GUI는 `@manta/core` 위의 동등한 adapter다.
- GUI는 CLI를 shell로 호출하지 않는다.
- GUI는 CLI stdout/stderr를 파싱하지 않는다.
- 로컬 파일 계약은 `@manta/core`가 소유한다.
- SQLite 인덱스 구현은 `@manta/core`의 무거운 런타임 의존성을 만들지 않도록 별도 모듈이나 adapter 경계에서 다룬다.

### Markdown은 원본, SQLite는 인덱스

Manta의 source of truth는 계속 로컬 Markdown 파일이다.
SQLite는 원본 DB가 아니라 파생 인덱스다.

SQLite가 다룰 수 있는 것:

- task id, title, status, created
- 파일 경로와 파일 해시
- 검색용 title/body 텍스트 인덱스
- 선택적 섹션 위치: `Intent`, `Notes`, `Decisions`, `Result`
- GUI 목록/검색/필터를 위한 캐시
- `manta context` 후보 추출을 위한 메타데이터

원칙:

- SQLite는 삭제해도 Markdown에서 다시 만들 수 있어야 한다.
- SQLite에만 존재하는 사용자 데이터는 두지 않는다.
- SQLite가 깨지면 `manta index rebuild`로 복구할 수 있어야 한다.
- Git으로 추적할 기본 자산은 Markdown 파일이다.
- SQLite는 Manta 프로그램을 써야 하는 이유를 만든다. 빠른 검색, 빠른 GUI, 빠른 context 조립은 단순 폴더 규약만으로는 약하다.

## 수요 근거

현재 근거는 시장 검증이 아니라 창업자 자신의 고통이다.

관찰된 워크플로:

1. Notion의 기획 문서를 읽는다.
2. 주로 티켓 번호를 얻기 위해 Jira 티켓을 만든다.
3. Notion 링크나 선택한 내용을 Codex에 붙여 넣는다.
4. 사용자 고유의 구현 방향을 추가하고 Codex에게 Markdown 구현 계획을 작성하게 한다.
5. 그 Markdown 계획을 바탕으로 Codex에게 구현을 요청한다.
6. 다른 세션에서 Markdown 계획을 다시 Codex에 전달해 계획과 코드가 맞는지 비교한다.
7. Markdown을 덜 기술적인 Jira 요약으로 다시 작성한다.
8. 티켓 번호가 붙은 Markdown을 개인 git 저장소에 백업한다.
9. PR/코드 리뷰 중 Markdown을 다시 사용한 뒤, 임시 태스크 문서를 삭제한다.

측정된 마찰:

- Notion에서 Codex로 옮기기: 5분, 실패하면 10분.
- Markdown 의도와 구현된 코드 비교: 약 5분.
- Jira용으로 내부 세부사항 제거: 3-5분.
- 백업과 정리는 거의 비용이 없다.

보통 태스크 하나당 13-20분의 반복 마찰이 생긴다.
더 날카로운 고통은 시간만이 아니다.
AI 세션이 바뀔 때마다 개발자가 태스크 의도를 다시 복원해야 한다는 점이다.

## 목표 사용자와 가장 좁은 진입점

목표 사용자:

> 혼자 일하는 개발자, 프리랜서, 인디 빌더, 또는 AI를 많이 쓰는 빌더. 이들은 작은 회사처럼 일하고, 하루 종일 AI 코딩 에이전트를 쓰며, 태스크 히스토리를 중요하게 보고, 가벼운 시스템을 원하며, 무거운 SaaS 워크플로보다 로컬 파일을 선호한다.

비목표 사용자:

- 이미 복잡한 Jira/Notion 워크플로에 깊게 묶여 있는 팀.
- 첫날부터 커스텀 필드, 권한 모델, 승인, 다부서 리포팅이 필요한 회사.
- Jira/Notion 동기화 유틸리티만 원하는 사용자.

가장 좁은 진입점:

> AI를 많이 쓰는 1인 빌더를 위한 local-first 태스크 시스템. 가벼운 Linear처럼 느껴지며, 이전 태스크를 미래 AI 세션의 컨텍스트로 다시 불러올 수 있다.

## 제품 spine

Manta는 다음 순서로 발전한다.

```text
Local Linear
→ Local Index
→ AI Context
→ Local GUI
→ Lightweight History
→ Import/Export
→ Pro Layer
```

### Phase 1: Local Linear

핵심 태스크 명령을 구현한다.

- `manta init`
- `manta add`
- `manta list`
- `manta show`
- `manta start`
- `manta done`
- `manta edit`
- `manta search`

상태는 폴더 위치가 결정한다.

```text
tasks/todo/
tasks/in-progress/
tasks/done/
```

task 파일은 가볍게 유지한다.

```markdown
---
id: task-1
title: ...
created: ...
---

자유 본문
```

### Phase 2: Local Index

각 프로젝트에 파생 SQLite 인덱스를 둔다.
이 인덱스는 source of truth가 아니라 재생성 가능한 작업 엔진이다.

초기 인덱스는 다음을 지원한다.

- 상태별 task 목록
- title/body 텍스트 검색
- 파일 변경 감지
- GUI 표시용 캐시
- `manta context` 입력 후보 추출

예상 명령:

```bash
manta index rebuild
manta index check
```

### Phase 3: AI Context

`manta context`는 과거 작업을 미래 AI 세션의 입력으로 다시 꺼내는 명령이다.

```bash
manta context task-1
manta context task-1 task-4 --max-chars 6000
manta context task-7 --for pr-review --max-chars 4000
```

v0 원칙:

- 모델을 호출하지 않는다.
- 네트워크를 사용하지 않는다.
- 로컬 task 파일만 읽는다.
- deterministic extractive output을 만든다.
- `--max-chars`는 문자 수 기준이다.
- 하나라도 task 조회에 실패하면 전체 실패한다.
- partial context는 출력하지 않는다.

### Phase 4: Local GUI

GUI는 무료 Local Workspace다.

기본 화면:

- 왼쪽: todo / in-progress / done task list
- 가운데: task Markdown editor
- 전역: command palette

AI context는 상시 오른쪽 패널이 아니라 on-demand action이다.

- `Copy AI Context`
- modal
- drawer
- command palette action

규칙:

- preview는 read-only다.
- 명시적 save action 없이는 파일을 변경하지 않는다.
- GUI는 별도 DB나 별도 source of truth를 만들지 않는다.

### Phase 5: Lightweight History

작업 파일 안의 선택적 섹션을 점진적으로 표준화한다.

```markdown
## Intent
## Notes
## Decisions
## Result
```

처음부터 복잡한 handoff packet을 강제하지 않는다.
섹션은 있으면 활용하고, 없어도 동작해야 한다.

별도 `manta doc` 명령은 보류한다.
source/reference 문서가 task 밖의 독립 제품 표면을 필요로 한다는 증거가 생기기 전까지 Docs phase는 만들지 않는다.

### Phase 6: Import / Export

Jira/Notion/GitHub는 source of truth가 아니라 bridge다.
Manta가 local source of truth라는 점이 분명해진 뒤 import/export를 추가한다.

### Phase 7: Pro Layer

무료 제품은 한 대의 로컬 머신에서 완결된 느낌이어야 한다.
Pro는 여러 기기, 깊은 검색, 고급 context intelligence, 외부 시스템 bridge가 필요해지는 순간부터 시작한다.

무료:

- CLI
- local file contract
- local task CRUD/search
- basic `manta context`
- Local Workspace GUI
- Markdown task editor
- local context preview
- Git-friendly backup

Pro 후보:

- encrypted continuity sync
- optional user-owned backup target
- semantic search
- advanced context compiler
- import/export connectors
- team/shared workspace later

Sync 원칙:

- 로컬 파일이 source of truth다.
- 클라우드는 원본 DB가 아니라 연속성 계층이다.
- 가능하면 암호화된 sync와 사용자 소유 백업 대상을 우선한다.

## 오류와 empty state 원칙

task 조회 실패는 구분한다.

```text
INVALID_TASK_ID
TASK_NOT_FOUND
DUPLICATE_TASK_ID
TASK_FILE_UNREADABLE
TASK_FILE_MALFORMED
```

`manta search` 빈 결과는 에러가 아니다.

```text
exit code: 0
stdout: No tasks matched "oauth".
```

`manta context`는 일부 task만 성공하는 partial output을 내지 않는다.

```text
exit code: 1
stderr: TASK_NOT_FOUND: task-999
stdout: 비어 있음
```

## 구현 순서

task-13 이후 순서는 다음과 같다.

1. `task-14`: `manta add`
2. `task-15`: `manta list` / `manta show`
3. `task-16`: `manta start` / `manta done`
4. `task-17`: `manta edit`
5. `task-18`: `manta search`
6. `task-19`: local SQLite index v0
7. `task-20`: `manta context` v0
8. `task-21`: Local Workspace GUI v0 기획

## 고객 검증 질문

AI를 많이 쓰는 1인 빌더, 프리랜서, 인디 개발자 5명과 이야기한다.
Manta를 먼저 피칭하지 않는다.

질문:

1. 지난 1주일 안에 AI 코딩 세션이 바뀌어서 작업 의도를 다시 설명한 적이 있나요?
2. 지금 task history는 어디에 남기나요?
3. 로컬 Markdown이나 개인 repo로 작업 기록을 남기나요?
4. 완료한 작업을 나중에 AI에게 다시 먹인 적이 있나요?
5. Codex나 Claude로 과거 작업을 재개할 때 가장 느린 부분은 무엇인가요?
6. 로컬 CLI와 무료 GUI가 있다면 지금의 개인 task history workflow를 대체할 수 있나요?
7. encrypted cross-device sync, user-owned backup target, semantic search, advanced context compiler, import/export connector 중 무엇에 돈을 낼 수 있나요?
8. 지금 이 workflow 문제를 해결하려고 돈을 내는 도구가 있나요?
9. 돈은 안 쓰더라도 task context를 직접 유지하는 데 매주 몇 분을 쓰나요?
10. 이 문제를 해결하려고 직접 만든 스크립트나 설치한 도구가 있나요?
11. 오늘 당장 돈을 내야 한다면 어떤 단일 기능 때문일까요?

## 성공 기준

초기 제품 성공:

- 사용자가 개인 티켓 번호 Markdown 백업 저장소를 Manta로 대체한다.
- 사용자가 터미널이나 GUI에서 매일 Manta를 사용한다.
- 사용자가 의도를 다시 설명하지 않고 task 파일에서 AI 코딩 세션을 재개할 수 있다.
- 완료된 이전 task를 10초 안에 이후 task의 context로 재사용할 수 있다.

비즈니스 검증:

- 최소 5명의 1인 빌더 또는 프리랜서가 이것이 자신의 AI 코딩 workflow와 맞는다고 동의한다.
- 최소 3명이 실제 업무에 일주일 동안 사용한다.
- 최소 1명이 sync, semantic search, advanced context compiler, import/export 중 하나에 돈을 낼 의향을 행동 기반으로 보여준다.

## 관찰한 점

- 원래 프레이밍은 Jira/Notion 연동에 가까웠지만, 더 강한 욕구는 source-of-truth ownership이다.
- 가장 명확한 사용자 표현은 "원본이야"와 "Linear 처럼 가벼워야해"였다. 이것이 제품 감각을 이끌어야 한다.
- 선택한 순서 `Local Linear -> AI Context -> Local GUI -> Lightweight History`는 제품이 방법론을 앞세우는 것이 아니라, 매일 쓰이는 효용을 먼저 만들고 그 효용에서 방법론을 끌어내야 한다는 뜻이다.
