# 12 Factor CLI Apps

CLI는 제품을 구축하는 환상적인 방법이다. 웹 애플리케이션과는 달리, 그것들은 구축하는 데 훨씬 적은 시간이 들고 훨씬 더 강력하다. 웹에서는 개발자가 프로그래밍한 것만 할 수 있다. CLI에서는 여러 도구들을 쉽게 서로 결합하여 고급 작업을 수행할 수 있다. 그것들은 사용하기 위해 더 많은 기술적 전문성을 요구하지만, 여전히 관리자 작업, 파워 유저 작업, 또는 개발자 제품에 잘 맞는다.

Heroku에서 우리는 12 factor app이라고 불리는 하나의 방법론을 만들어냈다. 그것은 유지보수하기 쉽고 훌륭한 웹 애플리케이션을 만들기 위해 설계된 원칙들의 집합이다. 같은 정신에서, 다음 CLI 애플리케이션을 만들 때 염두에 두어야 할 12개의 CLI 원칙이 있다. 이 원칙들을 따르는 것은 사용자들이 사랑할 CLI UX를 제공할 것이다.

또한 우리는 Node에서 훌륭한 CLI를 구축하기 위해 이러한 원칙을 따르도록 설계된 oclif라는 CLI 프레임워크도 구축했다.

---

## 1. 훌륭한 도움말은 필수다

CLI를 위한 좋은 도움말 문서를 갖는 것은 극도로 중요하다. UI로 사용자를 안내할 수 없기 때문에, 웹 애플리케이션을 구축할 때보다 훨씬 더 중요하다.

CLI는 CLI 내부 도움말과 웹 상의 도움말(README는 훌륭한 장소다)을 제공해야 한다. 이것은 터미널을 떠날 필요가 없는 즉시성을 제공하면서 동시에 Google이 사용자들을 도울 기회를 제공한다 (문서가 Google에 의해 색인되고 있는지도 확인하라).

이미 사용자들이 man 페이지를 원할 것이라는 것을 알고 있지 않는 한, 그것들을 출력하는 것은 굳이 신경 쓰지 않아도 된다. 그것들은 더 이상 충분히 자주 사용되지 않는다. 초보 개발자들은 그것들을 알지 못하고 Windows에서는 동작하지 않는다. 이미 CLI 내부 도움말이 있다면 오프라인 지원도 필요하지 않다. 그럼에도 불구하고, oclif에는 man 페이지 지원이 추가될 예정인데, 프레임워크에서는 의미가 있기 때문이다. 한 번 해결하면 모든 oclif CLI가 그 이점을 얻을 수 있다.

CLI에서는 다음의 모든 경우에 도움말이 표시되도록 해야 한다. 사용자가 무엇을 입력할지 제어할 수 없기 때문에, 이 모든 경우는 도움말을 보여야 한다.

```bash
# 모든 명령어 나열
$ mycli
$ mycli --help
$ mycli help
$ mycli -h

# 서브커맨드 도움말 얻기
$ mycli subcommand --help
$ mycli subcommand -h
```

`-h`, `--help`는 오직 도움말을 위해서만 예약된 플래그여야 한다. `$ mycli subcommand help`의 경우, help가 서브커맨드에 전달되어야 하는 인자일 가능성을 보장할 수 없다. 이 경우에는, 잘못된 인자 오류가 발생했을 때만 도움말을 보여주는 것이 더 낫다. 실제로 “help”라는 이름의 Heroku 앱이 존재하며, 이것이 나에게 이 문제를 일으킨 적이 있다.

Shell 자동완성은 사용자에게 도움을 제공하는 또 다른 훌륭한 방법이다.

도움말 자체에서는, 명령의 설명, 인자의 설명, 모든 플래그의 설명을 보여주고, 그리고 가장 중요하게는 CLI의 일반적인 사용 예시를 제공하라. 그것이 당신에게 명백해 보일지라도, 그것은 사용자들이 가장 자주 참고하게 되는 문서이다.

## 2. args보다 flags를 선호하라

CLI는 두 가지 타입의 셸 입력을 받을 수 있다: flags와 args이다. flags는 타이핑을 조금 더 요구하지만 CLI를 훨씬 더 명확하게 만든다. 예를 들어, Heroku CLI에서는 과거에 heroku fork라는 명령이 있었다. 그것은 복사할 소스 앱과 복사될 대상 앱을 입력으로 받았다. 처음에는 이것이 하나의 flag와 하나의 arg를 사용했다:

```bash
$ heroku fork FROMAPP --app TOAPP
```

하나의 flag와 하나의 arg를 사용하는 것은 어떤 것이 소스이고 어떤 것이 대상 앱인지 매우 혼란스러웠다. 우리는 이것을 둘 다 flag를 사용하는 방식으로 변경했다:

```bash
$ heroku fork --from FROMAPP --to TOAPP
```

이렇게 하면 어떤 것이 소스이고 어떤 것이 대상인지 명확하다.

참고로, 우리는 실제로 이 명령을 Heroku CLI에서 제거했지만, args가 얼마나 혼란스러울 수 있는지 보여주는 좋은 예시다.

하지만 때로는 args도 괜찮다. 예를 들어 `$ rm file_to_remove`처럼 인자가 명확한 경우에는 문제가 없다. 좋은 규칙은 인자 타입이 1개일 때는 괜찮고, 2개는 매우 의심스럽고, 3개는 절대 좋지 않다는 것이다.

길이가 가변적인 인자의 경우에는 여러 개를 사용하는 것이 괜찮다. (예: `$ rm file1 file2 file3`). 문제가 되는 것은 서로 다른 타입의 인자일 때다.

flags는 또한 자동완성 로직을 작성하기 훨씬 쉽다. 값이 무엇이어야 하는지 정확히 알 수 있기 때문이다.

다른 프로세스에 flags를 전달하는 CLI의 경우(예: heroku run), flag 파서는 `--` 인자를 받아야 하며, 그 이후의 모든 것은 단순히 인자로 전달되어야 한다. 이를 통해 다음과 같은 명령을 실행할 수 있다:

```bash
$ heroku run -a myapp -- myscript.sh -a arg1
```

좋다. 이어서 **3번부터 12번까지, 동일하게 직역(의역 없이 원문 구조 유지)**으로 번역한다.

## 3. 나는 어떤 버전을 사용하고 있는가?

다음 중 어떤 방법으로든 CLI 버전을 얻을 수 있도록 보장하라:

```bash id="z3a8kd"
$ mycli version
$ mycli --version
$ mycli -V
```

단일 명령 CLI가 아니고 `-v`, `--verbose` 플래그를 가지고 있는 경우가 아니라면, `$ mycli -v` 역시 단순히 CLI 버전을 표시하도록 하는 것이 좋다. 올바른 명령을 찾을 때까지 세 개의 다른 명령을 실행해야 하는 것은 매우 답답한 일이다.

버전 명령은 사용자에게 디버깅 정보를 요청할 때 주요하게 사용하는 곳이므로, 단순히 버전 번호뿐만 아니라 문제를 진단하는 데 도움이 될 수 있는 추가적인 유용한 정보들을 포함시키는 것이 좋다.

또한 (당신의 CLI가 어떤 API를 사용한다고 가정할 때) 서버 측 문제를 디버깅할 수 있도록 User-Agent로 버전 문자열을 전송하는 것을 권장한다.

## 4. 스트림을 신경 써라

stdout과 stderr는 사용자에게 메시지를 출력하는 동시에, 그 내용을 파일로 리다이렉트할 수 있도록 해주는 방법을 제공한다. 예를 들어:

```bash id="q7g3k1"
$ myapp > foo.txt
Warning: something went wrong
```

이 경고는 stderr에 있기 때문에 파일에 포함되지 않는다. 경고를 stdout에 출력하면 이 경우 경고가 보이지 않게 될 뿐만 아니라, JSON이나 바이너리 같은 구조화된 데이터에서는 특히 문제가 된다. 오류와 경고는 stderr를 사용하라. 기본적으로 stderr는 stdout이 리다이렉트되더라도 항상 화면에 출력된다.

stderr에 있는 모든 것이 오류는 아니다. 예를 들어 curl은 파일을 다운로드할 때 진행 상태 출력이 stderr에 있다. 이렇게 하면 stdout을 리다이렉트하면서도 진행 상태를 계속 볼 수 있다.

요약하면: stdout은 출력 데이터용이고, stderr는 메시지용이다.

CLI에서 서브커맨드를 실행하는 경우, 해당 서브커맨드의 stderr는 항상 사용자에게 전달되도록 해야 한다. 이렇게 하면 모든 문제가 최종적으로 사용자 화면에 표시된다.

## 5. 문제가 발생했을 때를 처리하라

CLI에서는 웹 애플리케이션보다 문제가 훨씬 더 자주 발생한다. UI로 사용자를 안내할 수 없기 때문에, 우리가 할 수 있는 유일한 것은 사용자에게 에러를 출력하는 것이다. 이것은 예상되는 동작이며 CLI 사용의 일부다.

무엇보다도, 에러 메시지를 유용하게 만들어라. 훌륭한 에러 메시지는 다음을 포함해야 한다:

* 에러 코드
* 에러 제목
* 에러 설명 (선택 사항)
* 에러를 해결하는 방법
* 더 많은 정보를 위한 URL

예를 들어, 파일 권한 문제로 CLI가 실패했다면 다음과 같이 표시할 수 있다:

```bash id="o2l7a8"
$ myapp dump -o myfile.out
Error: EPERM - Invalid permissions on myfile.out
Cannot write to myfile.out, file does not have write permissions.
Fix with: chmod +w myfile.out
https://github.com/jdxcode/myapp
```

모든 CLI가 이렇게 친절하다면 프로그래머가 되는 것이 얼마나 놀라운 일일지 생각해보라.

하지만 때로는 사용자가 예상하지 못한 오류를 만날 수도 있다. 이를 위해 전체 traceback 정보를 볼 수 있는 방법과, 환경 변수를 통해 전체 debug 출력을 볼 수 있는 방법을 제공하라.

oclif에서는 DEBUG 환경 변수가 설정되면 컴포넌트별로 그룹화된 debug 출력을 제공하는 debug 모듈을 사용한다. debug를 완전히 활성화하면 매우 많은 상세 로그를 제공하는데, 이는 문제를 디버깅할 때 매우 유용하다.

에러 로그는 사후 분석(post-mortem debugging)에 유용할 수 있지만, 타임스탬프를 포함해야 하고, 디스크 공간을 차지하지 않도록 주기적으로 잘라내야 하며, ANSI 색상 코드를 포함하지 않도록 해야 한다.

## 6. 화려하게 만들어라!

현대의 CLI는 자신을 드러내는 것을 두려워해서는 안 된다. 중요한 정보를 강조하기 위해 색상과 dimming을 사용하라. 오래 걸리는 작업을 보여주기 위해 spinner와 progress bar를 사용하라. 매우 오래 걸리는 작업이 끝났을 때 OS 알림을 활용하라.

하지만, 언제 기본 동작으로 되돌아가야 하는지도 알아야 한다. 사용자의 stdout이 tty에 연결되어 있지 않다면(일반적으로 파일로 파이핑하는 경우), stdout에 색상을 출력하지 마라 (stderr도 마찬가지다).

spinner와 progress bar 역시 tty가 아닐 때는 적합하지 않다. 이것들은 화면에서 덮어쓰기를 하기 위해 ANSI 코드를 출력하는 방식으로 동작하며, 이는 화면에서만 정상적으로 작동한다. 이러한 코드를 파일에 출력해서는 안 된다.

사용자는 이러한 화려한 출력이 필요 없을 수도 있다. 다음 조건에서는 이를 존중해야 한다:

* TERM=dumb
* NO_COLOR 설정됨
* `--no-color` 옵션 지정됨

또한 특정 CLI에서만 색상을 끌 수 있도록 `MYAPP_NOCOLOR=1` 같은 환경 변수를 제공하는 것도 좋다.

## 7. 가능하면 prompt를 사용하라

입력을 받을 때, stdin이 tty라면 사용자가 flag를 입력하도록 강제하기보다 prompt를 사용하라. 하지만 prompt를 필수로 요구해서는 안 된다. 사용자는 스크립트에서 CLI를 자동화할 수 있어야 하므로, 항상 prompt를 override할 수 있도록 해야 한다.

또 다른 좋은 사용 사례는 위험한 작업에 대한 확인이다. 예를 들어 Heroku 앱을 삭제할 때는 앱 이름을 다시 입력하도록 요구한다.

체크박스와 라디오 버튼은 사용자에게 선택지를 시각적으로 제공할 때 CLI UX를 향상시킬 수 있는 훌륭한 방법이다.

## 8. 테이블을 사용하라

테이블은 CLI에서 데이터를 출력하는 매우 일반적인 방법이다. 중요한 것은 출력의 각 행이 하나의 ‘엔트리’ 데이터여야 한다는 것이다. 테이블 테두리는 절대 출력하지 마라. 이는 매우 시끄럽고 파싱하기 어렵다.

각 행이 하나의 엔트리로 유지되면 다음과 같은 작업이 가능해진다:

* `wc`로 줄 수 계산
* `grep`으로 필터링

화면 너비를 고려하라. 기본적으로는 몇 개의 컬럼만 표시하고, `--columns` 옵션으로 추가 컬럼을 지정할 수 있게 하라.

현재 화면 너비를 넘는 행은 `--no-truncate`가 설정되지 않았다면 잘라내라.

기본적으로 컬럼 헤더를 표시하되, `--no-headers` 옵션으로 숨길 수 있게 하라.

`--filter`로 특정 컬럼을 필터링할 수 있게 하라.

`--sort`로 정렬을 지원하라 (역순 및 다중 컬럼 포함).

출력 형식으로 csv 또는 json을 지원하라. json은 구조화된 데이터를 출력하는 훌륭한 방법이며 jq로 처리할 수 있다. csv는 cut이나 awk와 함께 더 간단하게 사용할 수 있다.

## 9. 빠르게 만들어라

CLI는 빠르게 시작해야 한다. `$ time mycli`로 성능을 측정하라.

기준:

* 100ms 이하: 매우 빠름
* 100ms~500ms: 충분히 빠름 (목표)
* 500ms~2초: 사용 가능하지만 인상적이지 않음
* 2초 이상: 느림, 사용자들이 피하려 할 것

CLI가 큰 파일을 다운로드하거나 CPU 집약적인 작업을 수행하는 경우에는 느릴 수 있다. 이 경우 progress bar나 최소한 spinner를 보여주어야 한다. spinner만 있어도 체감 속도가 훨씬 빠르게 느껴진다.

## 10. 기여를 장려하라

코드를 오픈소스로 유지하라. 사용자가 직접 코드를 살펴보고 문제를 진단할 수 있게 한다. 다른 사람들에게도 도움이 되는 샘플이 되며, 조직의 이미지에도 좋다.

라이선스를 반드시 선택하라. GitHub와 GitLab은 CLI를 공개하기에 좋은 장소이며 README는 CLI 개요를 제공하기에 완벽한 장소다.

CLI를 로컬에서 실행하는 방법과 테스트 실행 방법을 문서화하라. 커밋 규칙, 코드 품질, 테스트 기준 등을 설명하는 기여 가이드를 제공하라.

코드 오브 컨덕트를 추가하라. 필요 없다고 느껴지더라도 일부 사람들에게는 중요하며, 문제가 발생했을 때 기준 문서로 활용할 수 있다.

oclif에서는 플러그인 시스템을 통해 CLI를 확장할 수 있다. 이러한 플러그인은 나중에 핵심 플러그인으로 포함될 수도 있다.

## 11. 서브커맨드를 명확히 하라

CLI에는 두 가지 유형이 있다:

* 단일 명령 CLI
* 다중 명령 CLI

단일 명령 CLI는 cp나 grep 같은 기본 UNIX 스타일 CLI다. 다중 명령 CLI는 git이나 npm처럼 서브커맨드를 첫 번째 인자로 받는다.

CLI가 단순하고 하나의 작업만 수행한다면 단일 명령 CLI가 적합하다. 하지만 대부분의 CLI는 서브커맨드를 사용하는 것이 더 유리하다.

사용자가 아무 인자도 전달하지 않았다면:

* 다중 명령 CLI: 서브커맨드 목록 출력
* 단일 명령 CLI: 도움말 출력

서브커맨드를 사용하다 보면 서브-서브커맨드가 필요해진다.

Git은 공백으로 구분한다:

```bash id="y9qk1z"
$ git submodule add ...
```

Heroku CLI는 콜론을 사용한다:

```bash id="7xmpg4"
$ heroku domains:add ...
```

콜론 방식이 더 좋다. 명령과 인자를 명확하게 구분해주기 때문이다.

기술적으로도 이유가 있다. 공백을 사용할 경우, 특정 인자가 서브커맨드인지 단순 인자인지 구분할 수 없는 문제가 발생한다.

## 12. XDG 규격을 따르라

XDG 규격은 파일을 어디에 저장해야 하는지 알려주는 훌륭한 표준이다.

환경 변수가 지정되지 않았다면 다음을 사용하라:

* 설정 파일: `~/.config/myapp`
* 데이터 파일: `~/.local/share/myapp`

캐시 파일:

* Unix: `~/.cache/myapp`
* macOS: `~/Library/Caches/myapp`
* Windows: `%LOCALAPPDATA%\myapp`
